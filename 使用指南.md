## 1. 系统概述

BridgeWatch AI 是一套面向监测中心的桥梁与隧道智能监测系统，具备以下核心能力：

| 能力 | 说明 |
|------|------|
| 视频风险识别 | 自动检测桥梁坍塌、桥面变形、车辆积压、桥面火灾四类事件 |
| 船撞融合预警 | 结合视频与 AIS/SHM 数据评估船舶碰撞风险 |
| 隧道异常监测 | 结合视频与环境传感器数据监测隧道安全 |
| 事件闭环管理 | 事件生成 → 告警通知 → 人工复核 → 归档 |
| 指挥大屏 | 总览面板展示实时风险态势和统计数据 |

## 3. 快速启动

### 3.1 一键启动（推荐）

项目根目录提供了快捷启动脚本：

```bash
# Windows — 双击 start-dev.bat 即可
# 该脚本会自动启动后端和前端

# 或分步启动：
start start-backend.bat   # 在终端1运行
start start-frontend.bat  # 在终端2运行
```

### 3.2 分步手动启动

#### 步骤 1：启动后端

```bash
cd backend

# 安装 Python 依赖
pip install -r requirements.txt

# 配置环境变量（首次需要）
cp .env.example .env
# ⚠️ 编辑 .env，将 DATABASE_URL 改为：
# DATABASE_URL=sqlite+aiosqlite:///./bridgewatch.db

# 启动后端服务（默认 http://localhost:8000）
python run.py
```

启动成功标志：终端显示 `Uvicorn running on http://0.0.0.0:8000`

#### 步骤 2：种子数据（可选）

系统在首次启动时会自动创建数据库表和默认用户，无需手动导入种子数据。

如需重置数据：

```bash
# 删除旧数据库后重启即可自动重建
del backend\bridgewatch.db  # Windows
rm backend/bridgewatch.db   # Linux/Mac

# 手动导入示例数据
cd backend
python -m app.seed
```

#### 步骤 3：启动前端

```bash
cd bridgewatch-ai

# 安装依赖
npm install

# 启动开发服务器（默认 http://localhost:5173）
npm run dev
```

启动成功标志：终端显示 `VITE ready in xxx ms` 和 `http://localhost:5173`

## 5. 各页面功能说明

### 5.1 总览指挥台（首页）

**功能**：综合展示系统运行态势。

页面包含 4 个区域：

| 区域 | 内容 | 数据来源 |
|------|------|----------|
| 顶部指标卡 | 高风险/中风险/低风险事件数 + 监测对象总数 | Dashboard API |
| 左侧面板 | 监测对象分布地图（示意） | — |
| 右侧面板 | 系统状态（事件数/告警数/桥梁数/隧道数） | Dashboard + Alert API |
| SHM 面板 | 桥梁振动/位移/水位监测数据 | Monitoring API |
| 底部三栏 | 船撞融合评分 / 隧道 CO/照度/车流 / 视频列表 | Fusion + Monitoring + Video API |

### 5.2 事件中心

**功能**：查看、检索和复核风险事件。

操作步骤：

```
1. 在搜索框输入关键字（桥梁名称/事件类型）筛选事件
2. 点击风险等级按钮（全部/高/中/低）快速筛选
3. 点击事件列表中某事件的「详情」按钮查看完整信息
4. 在右侧详情面板查看：
   - 事件截图（示意）
   - Sensor Trigger（触发传感器的信号描述）
   - Clip Window（事件发生的时间窗口）
   - Analyst Notes（AI 分析描述）
5. 点击「标记已复核」确认事件处理完毕
6. 点击「撤销复核」可取消确认
7. 点击「跳转回放」进入视频回放页面
```

### 5.3 视频回放

**功能**：查看事件相关的视频片段。

> ⚠️ 当前为演示界面，需对接真实视频流后才能播放实际画面。

### 5.4 普通桥梁

**功能**：查看普通桥梁四类风险识别结果。

页面展示：
- 四类事件（坍塌/变形/积压/火灾）的场景评估数据
- 模型能力雷达图
- 典型案例展示

### 5.5 船撞融合

**功能**：展示长大桥梁船撞融合预警分析。

页面展示：
- 船迹与桥区关系图（示意）
- 融合评分的构成因子（AIS 接近度/视频净距/应变峰值/振动异常）
- 桥区监测曲线（冲击/应变/净空）

### 5.6 隧道专题

**功能**：展示隧道环境监测与异常识别。

页面展示：
- 环境与交通走势图（CO 指数/照度/交通流量）
- 当班隧道异常事件列表

### 5.7 模型运维

**功能**：查看模型版本信息和推理任务状态。

| 区域 | 内容 |
|------|------|
| 模型版本 | 当前部署的模型名称、版本、状态 |
| 推理任务 | 任务列表（状态：排队中/运行中/成功/失败） |

### 5.8 报告中心

**功能**：生成和导出系统报告。

提供 3 种报告类型：
- **事件统计报告**：按类型、风险等级和时间维度统计
- **系统运行报告**：系统运行状态概况
- **数据导出**：导出事件数据为 Excel

### 5.9 系统设置

**功能**：用户管理和系统配置。

**用户管理**：
```
1. 查看用户列表（用户名/角色/状态）
2. 点击「添加用户」展开表单
3. 填写用户名、密码、显示名称、选择角色
4. 点击「创建」添加用户
5. 点击状态开关启用/禁用用户
6. 点击垃圾桶图标删除用户
```

**告警配置**：查看系统告警规则参数（只读展示）。
**通知设置**：查看通知渠道状态（邮件/SMS/页面通知）。
**系统信息**：查看系统版本和框架信息。

### 5.10 审计日志

**功能**：查看系统操作审计记录。

筛选方式：
- 按类型筛选：全部/登录认证/事件复核/用户管理
- 表格展示：时间、类型、级别、操作人、操作内容

## 6. 视频数据接入

### 6.1 视频文件存放位置

系统支持两种视频数据来源：

#### 方式一：本地视频文件

```bash
# 将视频文件放入以下目录
backend/data/files/videos/

# 支持的格式：.mp4, .avi, .mov
# 推荐分辨率：1920×1080
# 推荐时长：5-30 分钟
```

#### 方式二：视频流地址

在数据库中创建视频记录时指定 RTSP/RTMP 流地址：

```sql
-- file_url 字段填写流地址
INSERT INTO video_info (video_id, object_id, video_name, file_url) 
VALUES ('VID-xxx', 'OBJ-xxx', '实时流-01', 'rtsp://192.168.1.100:554/stream1');
```

### 6.2 手动导入视频

通过 API 导入视频：

```bash
# 需要有 admin 权限的 token
TOKEN=$(curl -s -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}' | \
  python -c "import sys,json; print(json.load(sys.stdin)['data']['access_token'])")

# 创建视频记录
curl -X POST http://localhost:8000/api/videos \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "object_id": "OBJ-20260508-001",
    "video_name": "南京长江大桥_巡检_20260520.mp4",
    "file_url": "/files/videos/南京长江大桥_巡检_20260520.mp4",
    "capture_time": "2026-05-20T10:00:00",
    "duration_seconds": 1800,
    "resolution": "1920x1080",
    "scene_type": "day"
  }'
```

### 6.3 自动触发推理

视频导入后，系统会自动处理：

```
视频入库 → InferenceWorker 轮询发现新任务
                 ↓
         执行 6 类检测器（桥梁/船撞/隧道）
                 ↓
         生成 EventRecord（事件记录）
                 ↓
         评估告警规则 → 生成 AlertRecord
                 ↓
         （可选）发送邮件通知
```

手动触发推理：

```bash
# 为已有视频创建推理任务
curl -X POST http://localhost:8000/api/tasks/inference \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"video_id":"VID-20260524-001","model_id":"MOD-20260508-001"}'
```

### 6.4 查看推理结果

```bash
# 查看推理任务状态
curl http://localhost:8000/api/tasks/inference \
  -H "Authorization: Bearer $TOKEN"

# 查看生成的事件
curl http://localhost:8000/api/events \
  -H "Authorization: Bearer $TOKEN"
```

## 7. 传感器数据接入

### 7.1 数据接入管线

系统提供了数据接入管线示例，支持模拟采集以下传感器数据：

| 传感器 | 数据类型 | 频率 | 接入方式 |
|--------|----------|------|----------|
| SHM 传感器 | displacement, vibration, strain | 每 5 分钟 | API / Modbus |
| AIS 接收器 | ais | 每 10 分钟 | 海事数据 API |
| 隧道环境传感器 | co, lux, traffic | 每 3 分钟 | Modbus / REST |
| 视频流 | — | 实时 | RTSP / 文件 |

### 7.2 手动触发数据采集

```bash
# 通过 API 触发一次采集
curl -X POST "http://localhost:8000/api/ingestion/trigger" \
  -H "Authorization: Bearer $TOKEN"

# 返回示例：{"shm": 3, "ais": 2, "tunnel": 3, "video": 1}
```

### 7.3 持续数据采集

```bash
# CLI 方式：每 60 秒采集一次
cd backend
python -m app.ingestion.cli --interval 60
```

### 7.4 对接真实数据源

编辑 `backend/app/ingestion/connectors.py`，将各 Connector 的 `collect()` 方法中的模拟数据替换为真实 API 调用即可。

## 8. 告警与通知

### 8.1 告警规则

系统内置以下告警规则：

| 触发条件 | 告警类型 | 严重等级 | 通知方式 |
|----------|----------|----------|----------|
| 新事件 risk_level=high | new_high_risk_event | critical | 页面 + 邮件 |
| 新事件 risk_level=medium | new_high_risk_event | warning | 页面 |
| 高风险 30 分钟未复核 | escalation | critical | 页面 + 邮件 |
| 中风险 60 分钟未复核 | escalation | warning | 页面 |

### 8.2 查看告警

- 顶部栏右方的 🔔 **铃铛图标** 显示未读告警数量
- 点击铃铛展开告警下拉面板
- 点击告警条目右侧的 ✅ **确认按钮** 标记为已处理

### 8.3 配置邮件通知

编辑 `backend/.env`，添加 SMTP 配置：

```bash
SMTP_HOST=smtp.qq.com          # QQ邮箱: smtp.qq.com
SMTP_PORT=587                   # 端口
SMTP_TLS=true
SMTP_USER=your@qq.com          # 你的邮箱
SMTP_PASSWORD=授权码             # QQ邮箱需使用授权码而非密码
SMTP_FROM=your@qq.com
SMTP_RECIPIENT=admin@example.com # 接收告警的邮箱
```

配置完成后重启后端，紧急告警（critical）会自动发送邮件。

## 10. 常见问题

### Q1：启动后端时提示端口被占用

```bash
# 查看 8000 端口占用
netstat -ano | findstr :8000

# 强制结束占用进程
taskkill /F /PID <进程ID>
# 或使用不同端口启动
python run.py --port 8001
```

### Q2：前端页面显示"演示模式"而非"后端已连接"

可能原因和解决方法：

```
1. 后端未启动 → 启动后端
2. 后端端口与前端代理不一致 → 检查 vite.config.ts 中的 proxy target
3. CORS 错误 → 检查后端 .env 中的 CORS_ORIGINS 是否包含前端地址
```

### Q3：登录提示"用户名或密码错误"

```
1. 确认使用正确的账号密码
2. 数据库可能未创建 → 删除 bridgewatch.db 后重启后端自动重建
3. 手动运行 seed: cd backend && python -m app.seed
```

### Q4：如何重置所有数据？

```bash
# 1. 停止后端服务
# 2. 删除数据库文件
del backend\bridgewatch.db
cd backend && python run.py
# 3. 系统会自动重建所有表
```

### Q5：推理任务一直显示 queued 状态

```bash
# 检查 Worker 是否在运行
# 启动后端时默认自动启动 Worker，终端会显示：
# "InferenceWorker started (poll_interval=5s)"

# 每 5 秒轮询一次 queued 任务，最长等待 5 秒即可看到状态变化
```

### Q6：如何导入大批量视频？

```bash
# 批量创建推理任务（会扫描所有已预处理但未推理的视频）
cd backend
python -m app.engine.batch --model-id MOD-20260508-001 --max-videos 50
```

### Q7：如何查看系统运行日志？

```bash
# 后端日志
docker-compose logs -f backend

# 前端日志（开发模式）
cd bridgewatch-ai
npm run dev  # 终端实时显示

# 审计日志（浏览器中查看）
# 左侧导航 → 审计日志
```
